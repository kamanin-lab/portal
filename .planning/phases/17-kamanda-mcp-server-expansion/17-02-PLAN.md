<rationale>
<!-- Why these are one-liners instead of the heavier `revenue-today.ts` style -->

`revenue-today.ts` is 260 lines because it does in-JS aggregation from raw WooCommerce REST v3 order responses (fetches orders + computes totals + splits by payment method + computes lagging metrics). The 5 new tools are different: their ability (on the WP side) has already done the aggregation in SQL and returned a pre-computed typed payload. The bridge client then validates that payload against a Zod schema.

So the tool builder has nothing to do except call the one matching client method. Keeping the builders as dedicated files (instead of inline in `mcp-server.ts`) serves three purposes:
  1. Mirrors the existing `revenue-today.ts` / `payment-attention.ts` import pattern — consistency.
  2. Keeps `mcp-server.ts` short even after Plan 03 adds 5 new `server.registerTool()` blocks.
  3. Leaves room for future enrichment (e.g. a caching layer between tool and bridge) without touching `mcp-server.ts`.
</rationale>
