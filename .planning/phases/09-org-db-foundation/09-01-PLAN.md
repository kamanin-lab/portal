<objective>
Phase 9 is a pure database migration phase against Cloud Supabase staging (`ahlthosftngdcryltapu`). It creates the full organisation schema (two tables, two helper SQL functions, FK columns on four tables, back-filled data, NOT NULL constraints, dual-mode RLS, and an extended notifications constraint) as a single atomic SQL migration. It contains a migration gate that rolls the transaction back on any data-integrity failure. No frontend or Edge Function code changes — existing portal functionality must continue working unchanged.

Purpose: Lay the data-layer foundation that Phases 10–13 will build on. The dual-mode RLS design guarantees zero breakage: old `profile_id = auth.uid()` policies remain active while new `organization_id IN (SELECT public.user_org_ids())` policies are added in parallel, so Edge Functions and frontend code that have not yet been migrated to org-scoped reads continue to work through `profile_id`.

Output:
- `supabase/migrations/20260414200000_org_foundation.sql` (atomic forward migration)
- `supabase/migrations/20260414200000_org_foundation_rollback.sql` (manual-only rollback reference)
- Staging database with `organizations`, `org_members`, helper functions, FK columns, migrated data, NOT NULL constraints, dual RLS, extended notifications constraint
</objective>

Follow the exact section order from CONTEXT.md (locked): tables → SQL functions → FK columns → data migration → NOT NULL constraints → RLS policies → notifications constraint extension → gate check DO block. Do NOT add explicit `BEGIN`/`COMMIT` — Supabase CLI wraps the file in its own transaction (per RESEARCH.md Pitfall 5).

### SECTION 1 — Tables

**public.organizations** (implements ORG-DB-01):
- `id uuid primary key default gen_random_uuid()`
- `name text not null`
- `slug text not null unique`
- `clickup_list_ids jsonb not null default '[]'::jsonb`
- `nextcloud_client_root text`
- `support_task_id text`
- `clickup_chat_channel_id text`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Enable RLS on `public.organizations` and attach `public.set_updated_at()` as a `BEFORE UPDATE FOR EACH ROW` trigger. Use `CREATE TABLE IF NOT EXISTS`.

**public.org_members** (implements ORG-DB-02):
- `id uuid primary key default gen_random_uuid()`
- `organization_id uuid not null references public.organizations(id) on delete cascade`
- `profile_id uuid not null references public.profiles(id) on delete cascade`
- `role text not null check (role in ('admin','member','viewer'))`
- `created_at timestamptz not null default now()`
- `unique (organization_id, profile_id)`

Enable RLS on `public.org_members`. Use `CREATE TABLE IF NOT EXISTS`.

Note: Phase 9 does not define Read policies for these two new tables; the Edge Functions that read them in Phase 10 will use the service role. No SELECT policies are required in this phase.

### SECTION 2 — SQL helper functions (ORG-DB-03, ORG-DB-04)

Create both with `SECURITY DEFINER`, `STABLE`, `SET search_path = ''`, fully-qualified names only. Use `CREATE OR REPLACE FUNCTION`.

`public.user_org_ids() returns setof uuid` — returns all `organization_id` values for the current `auth.uid()`. Returns empty set when `auth.uid()` is NULL (not an exception). Exact body per RESEARCH.md Pattern 1:
```sql
SELECT om.organization_id
FROM public.org_members om
WHERE om.profile_id = auth.uid()
  AND auth.uid() IS NOT NULL;
```

`public.user_org_role(org_id uuid) returns text` — returns the role text for the current user in the given org, or NULL if not a member. Exact body per RESEARCH.md:
```sql
SELECT om.role
FROM public.org_members om
WHERE om.organization_id = org_id
  AND om.profile_id = auth.uid()
LIMIT 1;
```

After creation, `REVOKE EXECUTE ... FROM public` and `GRANT EXECUTE ... TO authenticated, anon, service_role` on both functions (standard Supabase SECURITY DEFINER hardening).

### SECTION 3 — FK columns (ORG-DB-05)

All nullable, all idempotent:
```sql
ALTER TABLE public.credit_packages     ADD COLUMN IF NOT EXISTS organization_id uuid REFERENCES public.organizations(id);
ALTER TABLE public.client_workspaces   ADD COLUMN IF NOT EXISTS organization_id uuid REFERENCES public.organizations(id);
ALTER TABLE public.profiles            ADD COLUMN IF NOT EXISTS organization_id uuid REFERENCES public.organizations(id);

```

### SECTION 4 — Data migration (ORG-DB-06)

Implement exactly as RESEARCH.md Pattern 3 (DO block with FOR loop over `public.profiles`):
- For each profile, derive `org_slug := lower(regexp_replace(split_part(p.email, '@', 2), '\.[^.]+$', ''))`
- `INSERT INTO public.organizations (name, slug, clickup_list_ids, nextcloud_client_root, support_task_id, clickup_chat_channel_id) VALUES (COALESCE(p.company_name, split_part(p.email, '@', 1)), org_slug, COALESCE(p.clickup_list_ids, '[]'::jsonb), p.nextcloud_client_root, p.support_task_id, p.clickup_chat_channel_id) RETURNING id INTO new_org_id`
- `INSERT INTO public.org_members (organization_id, profile_id, role) VALUES (new_org_id, p.id, 'admin')`
- `UPDATE public.profiles SET organization_id = new_org_id WHERE id = p.id`
- `UPDATE public.credit_packages SET organization_id = new_org_id WHERE profile_id = p.id`
- `UPDATE public.client_workspaces SET organization_id = new_org_id WHERE profile_id = p.id`
- `UPDATE public.credit_transactions SET organization_id = new_org_id WHERE profile_id = p.id`

### SECTION 5 — NOT NULL constraints (ORG-DB-07)

```sql
ALTER TABLE public.credit_packages   ALTER COLUMN organization_id SET NOT NULL;
ALTER TABLE public.client_workspaces ALTER COLUMN organization_id SET NOT NULL;
```
`profiles.organization_id` stays nullable (backward safety per CONTEXT.md). `credit_transactions.organization_id` stays nullable (audit trail).

### SECTION 6 — Dual-mode RLS (ORG-DB-08)

Add new PERMISSIVE SELECT policies alongside the existing `profile_id = auth.uid()` policies. Do NOT drop or alter any existing policies.

```sql
CREATE POLICY "Users see org credit_packages" ON public.credit_packages
  FOR SELECT
  USING (organization_id IN (SELECT public.user_org_ids()));

CREATE POLICY "Users see org client_workspaces" ON public.client_workspaces
  FOR SELECT
  USING (organization_id IN (SELECT public.user_org_ids()));
```

The `(SELECT public.user_org_ids())` wrapper is mandatory — it triggers Postgres initPlan caching (per RESEARCH.md Pattern 2). Writing `public.user_org_ids()` directly would kill performance.

### SECTION 7 — Extend notifications_type_check (ORG-DB-10)

Per RESEARCH.md Pattern 4 (DROP + re-ADD in same transaction):
```sql
ALTER TABLE public.notifications DROP CONSTRAINT notifications_type_check;
ALTER TABLE public.notifications ADD CONSTRAINT notifications_type_check
  CHECK (type IN (
    'team_reply',
    'status_change',
    'step_ready',
    'project_reply',
    'project_update',
    'new_recommendation',
    'member_invited',
    'member_removed'
  ));
```

### SECTION 8 — Migration gate (ORG-DB-09)

Exact DO block per RESEARCH.md Pattern 5:
```sql
DO $$
DECLARE
  org_member_count integer;
  profile_count integer;
  null_list_count integer;
BEGIN
  SELECT COUNT(*) INTO org_member_count FROM public.org_members;
  SELECT COUNT(*) INTO profile_count FROM public.profiles;
  SELECT COUNT(*) INTO null_list_count
    FROM public.organizations
    WHERE clickup_list_ids IS NULL;

  IF org_member_count != profile_count THEN
    RAISE EXCEPTION 'Migration gate failed: org_members count (%) != profiles count (%)',
      org_member_count, profile_count;
  END IF;

  IF null_list_count > 0 THEN
    RAISE EXCEPTION 'Migration gate failed: % organizations have NULL clickup_list_ids',
      null_list_count;
  END IF;
END;
$$;
```

Note: the gate runs AFTER NOT NULL constraints are applied. Because `clickup_list_ids` has `NOT NULL DEFAULT '[]'::jsonb`, the null check is strictly a defensive assertion — any failure means something went wrong in section 4.

### SECTION 9 — Performance indexes (RLS efficiency)

```sql
CREATE INDEX IF NOT EXISTS org_members_org_id_idx ON public.org_members(organization_id);
```
