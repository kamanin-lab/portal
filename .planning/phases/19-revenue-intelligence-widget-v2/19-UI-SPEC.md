# Phase 19 — UI Design Contract

> Visual and interaction contract for the v2 Revenue Intelligence widget that replaces v1 `daily_briefing` at the same `ui://widgets/daily-briefing.html` URI. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Surface under design:** a sandboxed React 19 iframe rendered inside PORTAL's `RevenueIntelligencePage.tsx` via `<AppRenderer>` from `@mcp-ui/client`. The widget consumes 12 design tokens from the portal host via the `kmn/theme/*` postMessage bridge established in Phase 18. PORTAL itself sees **zero TypeScript diff** in this phase (PORT-04). All design below applies to code inside `G:/01_OPUS/Projects/mcp-poc/widgets/daily-briefing/src/`.
>
> **Constraints that dominate every decision below:**
> - Widget may reference ONLY the 12 tokens in `widgets/shared/widget-tokens.ts` — no 13th token without re-opening Phase 18 scope.
> - German-only user-facing strings (WIDG-QA-04).
> - Single-file bundle ≤ 300 KB gzipped; ≤ 80 KB gz target for Phase 19 code (WIDG-QA-05, D-19-02).
> - All charts hand-rolled (D-19-01) — no visx / recharts / victory / echarts.
> - All 4 blocks must render under 2s end-to-end (WIDG-QA-01).

## Spacing Scale

Declared values — multiples of 4 only, implemented as Tailwind utility classes:

| Token | Tailwind | px | Usage |
|-------|----------|-----|-------|
| xs | `gap-1` / `p-1` | 4px | Icon-to-label gap in attention rows, period-toggle label padding |
| sm | `gap-2` / `p-2` | 8px | Payment-method bar row, internal block-header-to-content gap |
| md | `gap-3` / `p-3` | 12px | Attention row internal spacing, heatmap legend gap |
| lg | `gap-4` / `p-4` | 16px | Block inner padding, inter-block space mobile |
| xl | `gap-6` / `p-6` | 24px | Block outer padding (desktop), block → block gap in 4-block stack |
| 2xl | `gap-8` / `p-8` | 32px | Widget root top/bottom padding |
| 3xl | `gap-12` | 48px | Not used in this phase (widget is vertically dense) |

### Exceptions

- **Heatmap cell gap:** 2px literal (`gap-[2px]`). A 4px gap would consume ~9% of the available grid width at 7×24 on typical portal widths and destroy the density signal. Approved exception — one cell boundary only.
- **Heatmap cell inner size:** cells use `aspect-square` with the 7×24 grid sized by `grid-template-columns: repeat(24, minmax(0, 1fr))`. Each cell ends up ~18-22px depending on the portal viewport width — not a spacing token, a derived value.
- **Heatmap row label column:** 40px wide (`w-10`) for `Mo Di Mi Do Fr Sa So` weekday abbreviations.
- **Heatmap hour axis row:** 20px tall (`h-5`) to hold `0..23` labels above the grid.
- **Attention sub-section divider:** 1px hairline (`border-t` with `--color-border`) between HeuteBlock main content and the `Zahlungsaufmerksamkeit` sub-section. Not a spacing value; a border.

### Block Layout

The 4 blocks render as a vertical stack inside the widget root — no horizontal grid, no 2-column layout. The widget inherits its width from the portal host (the `AppRenderer` iframe width), which at `max-w-4xl` in `RevenueIntelligencePage` gives ~768px desktop, ~100vw mobile.

```
root                 ← padding 32px top/bottom, 24px left/right
├─ HeuteBlock        ← gap-6 below
│  ├─ main content (run-rate + pace + payment-method bars)
│  └─ AttentionList  ← divider above, inset 16px, optional
├─ HeatmapBlock      ← gap-6 below
├─ RepeatBlock       ← gap-6 below
└─ BasketOrAovBlock  ← no trailing gap
```

## Typography

Font stack: `font-family: inherit` from portal — resolves to `"DM Sans", system-ui, ...` for body and `"DM Mono", ui-monospace, ...` for technical trio text (see `Technical trio` role below).

| Role | Size | Weight | Line Height | Tailwind | Where |
|------|------|--------|-------------|----------|-------|
| Display (primary KPI) | 28px | 600 | 1.15 | `text-3xl font-semibold leading-tight` | `Hochrechnung 4.280 €` (HeuteBlock), `31%` Wiederkaufrate (RepeatBlock), `847 €` Ø Bestellwert (BasketOrAovBlock aov_bands mode) |
| Heading (block title) | 18px | 600 | 1.3 | `text-lg font-semibold leading-snug` | `Heute`, `Bestellmuster — letzte 8 Wochen`, `Wiederkäufer — letzte 90 Tage`, and mode headers in BasketOrAovBlock |
| Body | 15px | 400 | 1.5 | `text-[15px] leading-normal` | `Bisher 1.240 €`, `Bei aktuellem Tempo`, plain-language basket gloss, attention row primary text |
| Caption / technical | 13px | 400 | 1.4 | `text-[13px] leading-snug` | Pace indicator (`vs. Ø gleiche Stunde letzte 7 Tage +18 %`), benchmark line (`Shopify B2C ~27 %`), AOV band labels, best-slot callout |
| Technical trio | 13px | 400 | 1.4 | `text-[13px] font-mono leading-snug text-[color:var(--color-subtle)]` | `Support 8% · Konfidenz 64% · Lift 3,2×` below basket pairs. ONLY monospace role in the widget. |

### Typography rules

- Exactly **5 type roles** (Display, Heading, Body, Caption, Technical trio) — no sixth. If a block needs an in-between size, the planner must round to one of these 5.
- **Exactly 2 weights:** `400` (regular) and `600` (semibold). No `500`, no `700`. `600` is reserved for Display + Heading only.
- **No `uppercase` transforms.** German content includes compound words and mixed casing (`Hochrechnung`, `Wiederkaufrate`) that must render as written. The only Unicode glyphs used (`↑ ↓ ▶ ▲ ▼ · ×`) are part of the copy, not typographic decoration.
- **No italic.** German text with italic is rare in merchant UI and conflicts with DM Sans's display intent.
- **Numbers in Display role:** `font-variant-numeric: tabular-nums` via `tabular-nums` class — prevents layout jitter during the count-up animation.
- **Count-up target display:** during animation, the numeric span renders the in-flight value (e.g. `3.142 €` → `3.789 €` → `4.280 €`) using Display role. No separate "animation size" — visual weight stays constant throughout the 600ms tween.

## Color

### 60 / 30 / 10 split

| Role | Token | Share | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `--color-bg` + `--color-surface` | ~60% | Widget root background (`--color-bg`), block card surface (`--color-surface`). Text sits on both. |
| Secondary (30%) | `--color-fg` + `--color-muted` + `--color-subtle` | ~30% | Text at 3 hierarchy levels (Display + Heading → `--color-fg`; Body → `--color-fg`; Caption / labels → `--color-muted`; technical trio / axis labels / empty heatmap cells → `--color-subtle`). |
| Accent (10%) | `--color-accent` | ~10% | Reserved list below — no other usage. |

### Accent reserved for

The accent color is used ONLY for:

1. Heatmap cell fill intensity (5-step scale from `color-mix(in oklch, var(--color-accent) 25%, transparent)` up to `var(--color-accent)` at 100%)
2. Period-toggle **active** state background (`[8 Wochen ✓]`)
3. Best-slot callout indicator (`█ Ihr bester Slot: Do 20:00`)
4. Positive trend arrow in RepeatBlock (`↑ +4 PP`) — when color is used, it is `--color-success`, not accent; accent is NOT used for trend indicators
5. Basket-block pair separator `+` between product names (rendered in `--color-accent` to visually bind the two names)
6. Count-up underline shimmer during the 600ms entry animation — a 1px `--color-accent` bar grows from left to right under the Display number, disappears at settle

Accent is **never** used for: block borders (use `--color-border`), body text (`--color-fg`), captions (`--color-muted`), attention-list urgency (use semantic colors below), payment-method bars (see below), AOV band bars (see below).

### Semantic colors (mandatory usage map)

Each semantic color is tied to a specific meaning; no overlap permitted.

| Semantic | Token | Used for | Example string |
|----------|-------|----------|----------------|
| Positive | `--color-success` | Pace indicator when `pace_vs_7day_avg_pct > 0`; trend arrow when `trend_pp > 0` in RepeatBlock | `+18 %` (green), `↑ +4 PP` (green) |
| Negative | `--color-danger` | Pace indicator when `pace_vs_7day_avg_pct < 0`; error skeleton primary text (`Daten nicht verfügbar` uses `--color-muted` not danger — see Copywriting); payment-failed urgency chip in AttentionList | `−7 %` (red), payment_failed chip background `color-mix(in oklch, var(--color-danger) 12%, transparent)` |
| Warning | `--color-warning` | Medium-confidence caption (`(Schätzung, geringe Datenbasis)`); invoice-overdue chip; on-hold chip | `(Schätzung, geringe Datenbasis)` (warning color, caption size) |
| Neutral | `--color-muted` | Pace indicator when the baseline is unusable — block shows `n/v` in muted rather than hiding (note: D-19-06 says hide the pace indicator; if ever shown as fallback, muted is the color) | n/a — normally hidden |

### Payment-method bars (HeuteBlock)

Three-way split visual for the paid payment methods (Klarna, PayPal, Stripe). Each row:

- Track: 8px tall, full width minus the label width, `--color-surface` background (slightly darker than block surface via 4% opacity accent tint: `color-mix(in oklch, var(--color-accent) 4%, var(--color-surface))`)
- Fill: variable `width: ${pct}%`, `--color-accent` at 100% opacity for the top-revenue method, 60% opacity for second, 30% opacity for third (determined by descending revenue order, not by method name). This gives visual priority to the top performer without introducing brand-specific colors for Klarna / PayPal / Stripe.
- Label: left-aligned, Body role, in `--color-fg`
- Percentage: right-aligned, Caption role, in `--color-muted`

### AOV band bars (BasketOrAovBlock, aov_bands mode)

Three bands (`< 500 €`, `500–1.500 €`, `> 1.500 €`). Each band renders two stacked mini-bars:

- **Share of count bar:** 6px tall, `--color-accent` at 50% opacity, `width: ${share_of_count_pct}%`
- **Share of revenue bar:** 6px tall, directly below, `--color-accent` at 100% opacity, `width: ${share_of_revenue_pct}%`
- The visual gap between the two opacities IS the insight ("17% of orders = 30% of revenue in the top band"). Do not pick different hues — the same-color / different-opacity contrast is the design.
- Band label (`< 500 €`): left-aligned, Caption role
- Twin percentages (`22% · 14% Umsatz`): right-aligned, Caption role; first number = share of count, second number with "Umsatz" suffix = share of revenue

### Heatmap intensity scale (HeatmapBlock)

Locked 5-step scale per cell, chosen from Claude's Discretion (D-19 says "planner picks"). Uses `color-mix(in oklch, var(--color-accent) N%, transparent)` for intensity — Tailwind v4 accepts this in inline `style={{background: ...}}` per RESEARCH.md Pattern 5:

| Intensity | Condition | Background |
|-----------|-----------|------------|
| 0 (empty) | `order_count === 0` | `var(--color-subtle)` at 12% opacity: `color-mix(in oklch, var(--color-subtle) 12%, transparent)` |
| 1 | 1st quintile of non-zero cells | `color-mix(in oklch, var(--color-accent) 20%, transparent)` |
| 2 | 2nd quintile | `color-mix(in oklch, var(--color-accent) 40%, transparent)` |
| 3 | 3rd quintile | `color-mix(in oklch, var(--color-accent) 60%, transparent)` |
| 4 | 4th quintile + top cell | `color-mix(in oklch, var(--color-accent) 85%, transparent)` — stops short of 100% so the best-slot callout (which renders at 100%) reads as distinctly brighter |

- Quintile boundaries computed on the non-zero buckets only (empty cells never pull quintile breakpoints toward 0).
- Best slot cell additionally gets a 1.5px `--color-accent` border at 100% opacity — visual emphasis beyond the color fill.
- `color-mix` fallback: no fallback layer shipped — staging Chromium is the only supported browser (Safari 15 edge case accepted per RESEARCH.md Pitfall 6).

### Color usage forbidden

- Hardcoded hex values in TSX (`#2B1878`) — must reference a token.
- `--color-danger` on purely-informational elements (e.g. "0 Bestellungen" is neutral, not danger).
- `--color-accent` on body text.
- Gradients. The 5-step scale + opacity mixing is the only color interpolation allowed.
- Shadow colors that differ from the block surface. If a block card has a shadow, it is `0 1px 2px rgba(0,0,0,0.04)` (small + subtle, no tint).

## Motion

Locked per D-19-03. The planner may only tune the values marked "discretion" below.

### 4-block entry stagger (first `toolResult` arrival only)

- **Pattern:** `AnimatePresence` + parent `variants={list}` with `transition: { when: "beforeChildren", delayChildren: stagger(0.08) }`; children `variants={item}` with `initial: {opacity: 0, y: 12}` → `animate: {opacity: 1, y: 0}`.
- **Stagger interval:** **80ms** (locked — was discretion; checked in dev harness against 60ms / 100ms; 80ms gives a clear "dashboard landing" cascade without making the 4th block feel slow to arrive).
- **Per-child duration:** 320ms with `ease: [0.22, 1, 0.36, 1]` (cubic-out).
- **TranslateY distance:** 12px — matches spec feel without feeling "heavy".
- **Trigger:** runs once, on the transition from `state: 'loading'` to `state: 'ok'`. Subsequent re-renders (period-toggle success, count-up, state mutations) do NOT re-stagger.

### Count-up (Display role numbers)

- **Applied to:** `Hochrechnung` € value (HeuteBlock), `Wiederkaufrate` % value (RepeatBlock), `Ø Bestellwert` € value (BasketOrAovBlock aov_bands mode only). No count-up on attention list amounts, no count-up on basket pair percentages.
- **Duration:** **600ms** (locked — was discretion; 400ms feels twitchy for large amounts like `4.280 €`, 800ms feels performative).
- **Driver:** `useMotionValue(0)` + `useSpring(motionValue, {stiffness: 120, damping: 20, mass: 0.8})` + `useMotionValueEvent(spring, "change", setDisplay)` per RESEARCH.md Pattern 4.
- **Formatting during animation:** every frame runs through `formatCurrency(display)` / `formatPercent(display)` so the user sees thousand-separator-correct intermediate values (`3.142 €` not `3142 €`).
- **Accent underline shimmer:** during the 600ms, a 1px `--color-accent` bar grows from `width: 0%` to `width: 100%` directly under the Display number, then fades to 0 opacity over 200ms. Stops when count-up settles.
- **Not retriggered:** count-up runs on the first render with a non-zero target. Period-toggle re-fetch (heatmap only) does NOT affect Display numbers. If `heatmap` data changes and the projected number stays the same (it would — projection lives in `run_rate` block), no animation fires.

### Heatmap container fade-in

- **Single tween:** the entire 7×24 grid wrapper fades from `opacity: 0` → `opacity: 1` over 400ms after the parent `HeatmapBlock` `motion.div` item variant has settled (so: ~320ms block entry + 400ms cell fade = 720ms total for the grid to be fully visible). No per-cell stagger. Complying with D-19-03 "container-level fade-in only".

### Period-toggle re-fetch feedback

- **During fetch:** the heatmap grid container (not the whole HeatmapBlock, not the period-toggle buttons) dims to `opacity: 0.6` over 120ms. Simultaneously a 14px circular spinner (`<div class="animate-spin rounded-full border-2 border-[color:var(--color-accent)] border-t-transparent">`) appears centered over the grid.
- **After fetch:** grid opacity restores to 1 over 200ms. Spinner removed. Best-slot cell callout re-renders with the new data.
- **Sibling blocks:** zero visual change. No shimmer, no dim, no re-stagger. Period-toggle is an atomic sub-block operation.
- **Button state during fetch:** the currently-clicked button gains `cursor-wait` + `disabled` + reduced opacity (`opacity-60`). The other two buttons remain interactive (user can cancel the pending fetch by clicking a different period, which starts a fresh fetch).

### Skeleton shimmer (BlockSkeleton variant='loading')

- **Locked implementation:** CSS-only animation (was discretion — Motion tween was the alternative; CSS is cheaper per KB and does not compete with the 4-block entry Motion budget).
- **Keyframes:** `@keyframes shimmer` with a 1.5s linear-infinite gradient sweep from `--color-bg` through `color-mix(in oklch, var(--color-subtle) 10%, transparent)` back to `--color-bg`, `background-size: 200% 100%`, `background-position` animated `0% 0%` → `-100% 0%`.
- **Shape:** the skeleton card matches each block's approximate content height (HeuteBlock ~220px, HeatmapBlock ~280px, RepeatBlock ~180px, BasketOrAovBlock ~240px — derived from the typical rendered heights of real data, verified in dev harness). `border-radius: var(--radius-lg)`.
- **No text.** The skeleton shows no "Lade…" label (the spinner + shimmer is the feedback).

### `prefers-reduced-motion` honoring

- `useReducedMotion()` from `motion/react` gates ALL Motion effects.
- **When reduced:**
  - 4-block entry becomes a single container-level fade-in (all 4 blocks fade together over 200ms, no stagger, no translate).
  - Count-up sets the final value immediately — no intermediate frames.
  - Heatmap container fade-in stays (it's a single tween anyway; 200ms OK).
  - Period-toggle dim + spinner stays (functional indicator, not decorative motion).
  - Skeleton shimmer stays (background-position animation is a single loop; users who opt out of reduce are accepting CSS background animations elsewhere in the portal).

### Anti-patterns (forbidden)

- Motion on individual heatmap cells (168 animated divs inside a sandboxed iframe stutter — locked by D-19-03).
- Stagger re-firing on period-toggle success (siblings must not re-stagger — D-19-05).
- Count-up on attention list amounts (would feel like a ticker; these are static money-at-risk values).
- Easing curves other than `[0.22, 1, 0.36, 1]` for block entry. Spring physics for count-up only.

**Imported only by HeuteBlock.** Renders the `payload.attention.categories` contents as a compressed list (D-19-04 "visually subordinate").

### `blocks/BlockSkeleton.tsx`

**Props:** `{ variant: 'loading' | 'error'; approxHeight: number }` where `approxHeight` is one of 180 / 220 / 240 / 280 (RepeatBlock / HeuteBlock / BasketOrAovBlock / HeatmapBlock respectively).

## Responsive Behavior

The widget sits inside the portal's `max-w-4xl` container (~768px on desktop, 100vw on mobile). The widget itself does not add media queries — it uses CSS flexbox + grid that naturally reflow. Specific behaviors at narrow widths:

| Element | < 480px behavior |
|---------|------------------|
| 4-block stack | Stays vertical. `gap-6` between blocks kept. Block inner padding reduces from `p-6` to `p-4` via Tailwind responsive prefix `p-4 sm:p-6`. |
| HeuteBlock pace + reference rows | Stay horizontal (label left, value right). Values remain Caption size. |
| HeuteBlock payment-method bars | Stay horizontal (label + bar + percent on same line). If a payment-method title is long (>16 chars) it truncates with `text-ellipsis`. |
| HeatmapBlock 7×24 grid | Cells get tiny (~10px square at 360px width) but stay square via `aspect-square`. Hour axis labels use `text-[10px]` below 480px — readability degrades but the visual pattern survives. |
| HeatmapBlock period-toggle | Buttons wrap to 2 rows if needed (`flex-wrap`). |
| BasketOrAovBlock basket pair Line 1 | Product names wrap if they exceed line width — the `+` separator stays with the first product name (no orphaned `+` on a new line). |
| BasketOrAovBlock AOV band rows | Band label left + twin percentages right stay on the same line; bars below take full row width. |
| AttentionList rows | On < 360px, the right column (amount + Öffnen) may wrap below the left column. Acceptable — attention list is a triage surface, not a primary read. |

Portal viewport targets (per portal conventions):
- Desktop: ≥ 1024px
- Tablet: 768–1023px (portal bottom nav hides; widget gets full inline width)
- Mobile: < 768px (portal bottom nav shows; widget is narrower)

The widget has no viewport-specific code paths; it relies purely on the flex/grid layout + responsive Tailwind utilities.
