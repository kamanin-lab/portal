# Phase 11 — UI Design Contract

> Visual and interaction contract for Phase 11: org-frontend-auth.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase Character

This is a **data-layer and context wiring phase**. No new pages are introduced. No new visual surfaces are created. The only observable UI changes are:

1. Three action elements become conditionally absent for viewer-role users (`NewTicketDialog` trigger, `CreditApproval`, `TaskActions`).
2. Existing surfaces (tickets, workspaces, credits) silently switch their data source from `profile_id` to `organization_id` — visually transparent.

The design contract governs the absence states and loading behavior, not new components.

## Viewer Role — Absence Contract

This is the primary visual interaction this phase introduces. Three elements become absent (rendered as `null`) when `isViewer === true`:

| Element | Location | Guard Pattern | Viewer sees |
|---------|----------|---------------|-------------|
| "Neue Aufgabe" button / `NewTicketDialog` trigger | `TicketsPage.tsx` (the trigger button, not inside the dialog) | `const { isViewer } = useOrg(); if (isViewer) return null` | Nothing — no button, no placeholder, no disabled state |
| `CreditApproval` | `TaskDetail.tsx` → `CreditApproval.tsx` | Same inline guard | Nothing |
| `TaskActions` | `TaskDetail.tsx` → `TaskActions.tsx` | Same inline guard | Nothing — task info and status badge remain visible; only action buttons are absent |

**Critical contract:** Absence must be **complete removal**, not disabled/greyed-out state. Viewers see task information, status, and comments — they just cannot act. Do not render a disabled button or a tooltip explaining why it is absent. `return null` only.

## OrgContext Loading State Contract

`OrgProvider` fetches org data after `user` becomes available. During this fetch:

| State | UI Behaviour |
|-------|-------------|
| `isLoading: true` (org fetch in flight) | Render `null` for the three guarded elements (same as viewer outcome); or pass `isLoading` down and render `<Skeleton>` on the credits balance display only |
| Org fetch succeeds | Normal render — role determines visibility |
| Org fetch fails (network error) | Silent fallback per CONTEXT.md D-04: treat as `orgRole: 'member'`, `isAdmin: false`, `isMember: true`, `isViewer: false` — all UI visible |
| Legacy user (no `org_members` row) | Same silent fallback as fetch-fail above |

**Loading skeleton contract (credits balance only):**
When `useCredits` transitions from `profile_id` to `organization_id` query key, there is one re-fetch on mount. Use `<Skeleton className="h-4 w-16 rounded" />` (existing `skeleton` shadcn component) in place of the credit balance number only. No skeleton for buttons or structure.

## Component Interaction Patterns

### OrgProvider state shape

Mirrors `AuthProvider` exactly. Same pattern:
- `useState` for org data + loading flag
- `useEffect` triggered by `user` from `useAuth()`
- Context value provided to tree below

No React Query for the OrgProvider fetch itself — provider-level state (per CONTEXT.md "Specific Ideas" section). React Query is used inside `useCredits` and `useWorkspaces` after org is available.

### React Query cache keys

| Hook | Old key | New key |
|------|---------|---------|
| useCredits (package) | `['credit-package', user?.id]` | `['credit-package', organization?.id]` |
| useCredits (balance) | `['credit-balance', user?.id]` | `['credit-balance', organization?.id]` |
| useWorkspaces | `['workspaces', user?.id]` | `['workspaces', organization?.id]` |

All queries gated with `enabled: !!organization?.id`.
