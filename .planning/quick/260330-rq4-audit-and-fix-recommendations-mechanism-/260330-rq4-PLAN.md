In the cache update block (lines 649-662), add tag clearing for decline_recommendation. Currently the cache update only writes `status`, `status_color`, `last_synced` (and `due_date` for accept). The `recommendation` tag stays in task_cache until the webhook fires.

After the existing cache update at line 658-662, add a secondary update specifically for decline_recommendation that removes the recommendation tag from the task_cache `tags` JSONB column:

```typescript
if (action === "decline_recommendation" || action === "accept_recommendation") {
  // Remove recommendation tag from cached tags so UI doesn't show stale block
  const { data: cachedTask } = await supabase
    .from("task_cache")
    .select("tags")
    .eq("clickup_id", taskId)
    .eq("profile_id", userId)
    .maybeSingle();

  if (cachedTask?.tags) {
    const rawTags = cachedTask.tags as Array<{ name: string }>;
    const filteredTags = rawTags.filter((t: { name: string }) => t.name.toLowerCase() !== "recommendation");
    await supabase
      .from("task_cache")
      .update({ tags: filteredTags })
      .eq("clickup_id", taskId)
      .eq("profile_id", userId);
  }
}
```

This ensures that when the frontend refetches after invalidateQueries, the task_cache row no longer has the recommendation tag, so `isRecommendation` evaluates to false. This is a belt-and-suspenders fix alongside the portalStatus exclusion -- both fixes independently prevent the block from reappearing.

Note: Apply this for BOTH accept and decline to make the accept flow more robust too (currently accept relies on onClose hiding the sheet, but if the user re-opens the task before the webhook fires, they would see the block briefly).
